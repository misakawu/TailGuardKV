## 一句话讲清

> 现有 KV 压缩和分层系统主要优化平均质量、平均延迟或固定 profile，却不能回答“这一条请求采用近似 KV 后，质量损失超过阈值的概率是多少”。TailGuardKV 为每个请求和缓存 profile 产生经校准的质量风险上界，在显存与 TTFT 压力下只选择满足 \((\varepsilon,\delta)\) 质量 SLO 的压缩/剪枝动作；高风险对象退回无损 offload、恢复或重算，从而在控制质量违例率的前提下降低 p95 TTFT 和显存峰值。

## 重新审核后的核心判断

| 旧主张 | 审核结论 | 新处理 |
| --- | --- | --- |
| LPE 比 LRU/LFU 更好 | H1 五个 budget 全部失败，且 score 在同质 block 上退化为 \(p_{reuse}\) | 删除独立 LPE；换出默认使用 LRU/引擎策略 |
| 精度 × 放置联合调度具有首创性 | AdaptCache 已做逐 entry 压缩与设备放置；KVServe 已做质量/SLO 感知在线 profile 控制；TTKV 已做异质精度分层 | 作为系统动作空间保留，不再作为创新 |
| restore-vs-recompute 临界带宽可作为核心定理 | 本质是直接代价比较，难支撑 CCF-A 核心理论 | 降为控制器中的无损 fallback 规则 |
| 在线 regret 上界可作为第二核心 | 与实际系统动作、观测和损失定义结合不够紧 | 删除，避免为了理论而理论 |
| 质量预算 \(\varepsilon\) 可直接使用 | 原方案没有在线可观测、可校准的质量损失 | 改为 \((\varepsilon,\delta)\) 尾部质量 SLO，并用 conformal risk control 校准 |

## 新贡献结构

1. **问题与指标**：把 KV 管理定义为带质量违例率约束的在线控制问题，而不是平均质量与平均延迟的加权和。
2. **风险校准器**：学习请求/profile 的质量损失并输出分布无关的风险上界；按任务与长度分组校准，报告真实 coverage。
3. **安全控制器**：在风险安全集合内选择压缩、剪枝、GPU/CPU 放置或精确重算；风险不安全时自动回退到无损动作。
4. **漂移闭环**：通过少量 shadow audit 更新校准阈值，使工作负载变化后质量违例率重新回到 SLO。

## 本题边界

- 复用 vLLM / LMCache / KIVI / H2O / SnapKV 的机制，不重写 serving engine 和压缩 kernel。
- 不把 LRU、offload、量化或剪枝本身包装成创新。
- 不承诺每条请求确定性无损；目标是明确的边际或分组违例率控制，并诚实说明 conformal 保证的假设。
- 预实验阶段就实现完整 QRP/CG/STC、baseline adapter 和统一 runner；H0/H1/H2-lite 通过后才扩大模型、任务、网格和设备规模。

## 0.1 方向重构结论

截至 2026-07-14，原 EdgeKVTiers 的核心叙事已经不够新：

- [AdaptCache](https://arxiv.org/abs/2509.00105) 已经对每个 KV entry 联合决定压缩算法、压缩率和 DRAM/SSD 放置，并使用 task-specific quality estimator。
- [KVServe](https://arxiv.org/abs/2606.18456) 已经根据 workload、带宽、SLO 和质量预算在线选择 KV compression profile。
- [TTKV](https://arxiv.org/abs/2604.19769) 已经做 HBM/DRAM 的异质精度 temporal tiering。
- [MorphServe](https://arxiv.org/abs/2506.02006) 和 [SparKV](https://arxiv.org/abs/2604.21231) 已经覆盖边缘/受限设备上的动态 KV 大小与精度适配。
- H1 又证明同质 vLLM block 上的 LPE score 退化且稳定劣于 LRU。

因此，继续声称“首次联合精度、生命周期和放置”会被审稿人直接用最新竞品否定。新的文献空白必须从“动作更多”转向“约束更可信”：

> **当前系统大多报告平均任务质量或用离线 profile 作为质量预算，却缺少对单请求/任务组尾部质量违例率的显式、校准和可审计控制。**

[The Pitfalls of KV Cache Compression](https://arxiv.org/abs/2510.00231) 说明平均质量可能掩盖特定指令的灾难性退化；[The Risk of KV Cache Compression](https://arxiv.org/abs/2607.01520) 进一步把压缩风险形式化。这使“tail-quality SLO”比旧的平均质量预算更有现实必要性。

## 0.3 最强竞品拆解

### AdaptCache

**已经做了什么**：逐 entry 选择压缩算法、压缩率和设备放置；离线 estimator 预测 task quality；在线根据复用、延迟、存储和质量 utility 贪心决策。

**不能再声称什么**：不能再把“per-entry quality-aware compression + placement”作为首创。

**仍可区分之处**：AdaptCache优化质量加权 utility，主要评估平均任务质量；TailGuardKV优化显式违例率 \(\Pr[\ell>\varepsilon]\le\delta\)，并提供校准、审计和无损回退。

### KVServe

**已经做了什么**：离线 profiling compression profile，在线根据 workload、bandwidth、SLO、quality constraint 选择 profile，并用 bandit 适配。

**不能再声称什么**：不能把“在线根据质量预算选 profile”作为新意。

**仍可区分之处**：KVServe的约束主要建立在 service/profile 统计上；TailGuardKV必须证明请求级或分组 tail risk 的 calibrated coverage，并报告分布漂移后的违例恢复速度。

### TTKV / MorphServe / SparKV

这些工作共同说明“异质精度、动态压力、边缘适配”本身已不够新。本题只能把它们当动作空间和 baseline，贡献必须落在**风险定义、校准机制和安全 fallback**。

## 0.4 新问题的必要性

平均质量无法代替质量 SLO。两个策略都可能平均只下降 1%，但策略 A 的所有请求都下降约 1%，策略 B 可能是 95% 请求无损、5% 请求完全错误。对问答、代码和长会话，后者更危险。

本题定义：

- $\varepsilon$：单请求可接受质量损失阈值。
- $\delta$：允许超过阈值的最大比例。
- $\ell_i(a)$：请求 \(i\) 使用 profile \(a\) 相对 full-KV 的归一化质量损失。
- 质量 SLO：$Pr[\ell_i(a_i)>\varepsilon]\le\delta$。

系统只能在满足风险约束的动作中优化 p95 TTFT 和显存；如果近似动作都不安全，则选择 full-KV offload、恢复或重算。

## 0.7 一体化预实验：直接实现算法并与 baseline 比较

预实验不是单独的问题验证脚本，而是**缩小规模的主实验**。必须直接交付可继续扩展的 TailGuardKV 最小完整原型：

### 0.7.1 一次性实现的代码骨架

```text
tailguardkv/
  profiles/        # full、KIVI、H2O/SnapKV、exact-offload
  policies/        # TailGuardKV、static、utility、uncalibrated
  calibration/     # predictor、simultaneous conformal、policy-level CRC
  backends/        # measured-replay backend + vLLM/LMCache adapter
  metrics/         # quality、coverage、TTFT、memory、overhead
  configs/         # pilot/main 共用，主实验只扩大参数网格
  run_experiment.py
```

所有策略统一实现：

```python
class Policy:
    def decide(self, request, cache_state, device_state) -> Action:
        ...
```

baseline 与 TailGuardKV 必须经过同一 backend、同一 trace、同一指标代码，禁止为不同策略写不同实验脚本。

### 0.7.2 Pilot 配置

| 项目 | 最小配置 |
| --- | --- |
| 模型 | Qwen2.5-7B-Instruct |
| 任务 | 1个 + 摘要/代码任选1个；每类先200–300请求 |
| Profile | full、KIVI-4bit、KIVI-2bit、H2O/SnapKV一档、exact fallback |
| 显存预算 | 2个真正触发profile选择的紧预算 |
| 质量SLO | $\varepsilon$ 2档，$\delta\in\{0.05,0.10\}$ |
| 重复 | 系统回放每cell至少3次 |

质量 loss、profile latency 和 memory 必须来自实际模型/profile实测，不能使用拍脑袋的解析常数。预实验可以先用 measured-replay backend 做大网格，但必须完成一条 vLLM/LMCache 小 trace 的 smoke test，保证接口能落到真实引擎。

### 0.7.3 必须同时实现的 baseline

1. `full_lru`：全精度 + 引擎默认换出。
2. `static_best`：离线选择满足平均质量约束的最佳固定 profile。
3. `static_safe`：满足 tail SLO 的最保守固定 profile。
4. `utility_dynamic`：AdaptCache 风格平均 quality-utility 控制器。
5. `uncalibrated_dynamic`：使用点预测阈值、不做 conformal。
6. `tailguard`：QRP + simultaneous/policy-level calibration + STC + exact fallback。
7. `quality_oracle`：仅作上界，不参与“最佳可部署 baseline”排名。

### 0.7.4 一次运行同时回答 H0/H1/H2-lite

同一批 pilot 结果必须输出：

- H0：mean/p95/p99/CVaR/violation，证明 tail 问题。
- H1：target-vs-empirical coverage、worst-group、UCB slack、exact fallback ratio。
- H2-lite：固定 ($\varepsilon,\delta$)下各策略的 p95 TTFT、显存和动作分布。
- 系统开销：predictor、calibration、controller 和迁移时间。

### 0.7.5 Pilot 通过标准

必须同时满足：

1. H0 tail-risk 成立。
2. $\delta=5\%$ 时 empirical violation ≤8%，且 exact fallback ratio <60%，避免“全部full”的空保证。
3. 至少2个有效cell中，相比最佳可部署 baseline，p95 TTFT降≥10%或显存降≥15%。
4. controller开销 <5% TTFT。
5. `run_experiment.py` 只修改配置即可扩展到主实验。

通过后，主实验只做四件事：增加模型、增加任务、增加预算/SLO网格、切换到真实 vLLM/LMCache/端侧 backend。算法和 baseline 不得推倒重写。

## 0.8 第一阶段 Go/No-Go

绝对截止日期：**2026-09-15**。

- **Go 主实验**：一体化 Pilot 的五项门槛全部通过，直接扩大 E3 主结果矩阵。
- **延长两周**：tail-risk 与coverage通过，但收益为5%–10%，且 Quality Oracle gap ≥20%。
- **No-Go**：H0无尾部问题、coverage只能靠全部full、或对baseline没有系统收益信号。

第一阶段交付物必须包括：

- 可运行的 `run_experiment.py` 和统一 Policy/Backend 接口。
- TailGuardKV 与至少5个 baseline 的同表对比。
- 每请求质量损失数据、完整 trace 和配置。
- motivation、coverage-efficiency、p95-memory 三类图。
- vLLM/LMCache smoke test。
- 一页 Go/No-Go 决策，不用“算法已实现”代替收益门槛。

# 一章 — 论文整体工作介绍
---

## 1.1 背景

KV cache 压缩、剪枝和分层存储已经能显著降低显存与传输成本，但大多数系统用平均准确率、平均 perplexity 或离线 profile 表示质量。真实部署更关心另一件事：某个近似 profile 是否会让少数请求产生不可接受的质量崩溃。

边缘设备显存小、带宽和负载变化快，更容易频繁切换压缩率、放置层和重算动作。如果控制器只优化平均质量或固定 utility，系统可能在平均指标好看的同时违反单任务或长上下文请求的质量 SLO。

## 1.2 研究问题

> 在显存预算 \(M\)、延迟 SLO 和工作负载漂移下，如何为每个可复用 KV 对象选择精确、量化、剪枝、offload 或重算动作，使 p95 TTFT 最低，同时控制质量损失超过 \(\varepsilon\) 的比例不超过 \(\delta\)？

这包含三个子问题：

1. 决策发生时没有 full-KV 输出，如何估计近似动作的尾部质量风险？
2. 如何把统计风险上界转化为在线可执行的 profile 安全集合？
3. workload shift 后，如何用很少的 full-reference audit 恢复 coverage？

## 1.3 系统概述

TailGuardKV 由四个逻辑模块组成：

| 模块 | 职责 |
| --- | --- |
| Quality Risk Profiler（QRP） | 提取请求、缓存对象和 profile 特征，预测质量损失 |
| Conformal Guard（CG） | 将点预测校准为分组风险上界，生成安全 profile 集合 |
| Safe Tier Controller（STC） | 在安全集合内选择精度、剪枝率、GPU/CPU 放置和 exact fallback |
| Audit & Adaptation Loop（AAL） | 抽样 shadow full-KV，更新校准阈值并检测漂移 |

换出顺序不再设计独立 LPE。控制器决定“使用哪个安全 profile”；同 profile 内的普通缓存淘汰直接采用 LRU/引擎默认策略。

## 1.4 贡献

1. **Tail-quality SLO 问题定义与评测协议**：同时报告平均质量、q95/q99 loss、违例率、worst-group risk 与 p95 TTFT。
2. **经校准的 KV 近似风险守门器**：把 profile 质量预测转化为有限样本风险上界，避免把不确定的点预测当作保证。
3. **风险安全的缓存动作控制器**：只在安全 profile 中优化系统成本，并以无损 offload/恢复/重算保证可行 fallback。
4. **漂移下的低审计自适应闭环**：用少量 shadow audit 维持长期 coverage，量化恢复速度和审计成本。

## 1.5 不作为贡献的内容

- KIVI/H2O/SnapKV 的 kernel。
- vLLM/LMCache 的缓存与传输机制。
- LRU、LFU、PGDSF 或普通 offload。
- “压缩率越高越省显存”这类直接观察。
- restore 与 recompute 的简单代价比较。

## 1.6 论文成立的最小条件

本题不是“做完系统就能投”。必须同时满足：

- H0：tail-quality 问题真实且平均指标确实掩盖风险。
- H1：风险校准既守住违例率，又不退化为全部使用 full-KV。
- H2：在同一质量 SLO 下，相比最强 baseline 有显著系统收益。
- H4：真实引擎与受限设备上收益仍成立。

H0/H1/H2-lite 不再按三个独立阶段串行执行，而是在同一个端到端 Pilot 中同时测量：算法先完整实现，再由同一批 TailGuardKV-vs-baseline 结果判断问题、保证和收益。任何一条核心门槛失败，都要按十章退出或改题。

# 四章 — 主要创新点
---

## 4.1 C1：尾部质量 SLO 的 KV 管理问题

现有方法常见目标是:
			$\min \text{Latency}+\lambda\cdot\mathbb{E}[\text{QualityLoss}]$

本题改为：
			$\min \operatorname{CVaR}_{0.95}(\text{TTFT})\quad\text{s.t.}\quad\Pr[\ell_i(a_i)>\varepsilon]\le\delta,\; M_t\le M$

创新不在公式形式本身，而在于建立完整系统闭环：每请求 loss 数据、可决策特征、风险校准、资源控制、audit 和真机验证。

**强度：中强，前提是 H0 证明 tail 与平均质量存在实质差异。**

## 4.2 C2：Calibrated Quality Guard

QRP 先预测 \($\hat\ell(x,a)$\)，CG 再利用校准残差得到风险上界 \(U(x,a)\)。动作只有在 \($U(x,a)\le\varepsilon$\) 时才进入安全集合。因为控制器会在多个 profile 中自适应选择，不能直接复用单 profile coverage；必须采用 simultaneous calibration（为各动作分配 \($\delta_a$\)，满足 \($\sum_a\delta_a\le\delta$）或在 held-out trace 上直接校准完整策略。

为避免整体 coverage 掩盖局部失败，按任务类型、上下文长度和 profile 做 Mondrian/group calibration；分布变化时由 audited 样本更新阈值。

**与竞品差异**：

- AdaptCache：点估计进入 utility，本题把不确定性校准成显式违例约束。
- KVServe：service/profile 级质量预算，本题报告请求级损失分布与分组 coverage。
- Pitfalls/Risk：揭示风险或给压缩算法，本题将风险控制接入在线 serving。

**强度：强，决定论文能否达到 CCF-A。**

## 4.3 C3：Risk-Safe Tier Controller

对每个缓存对象，动作集合可以包括：${\text{full@GPU},\text{quant@GPU},\text{pruned@GPU},\text{full@CPU},\text{restore},\text{recompute}}$

控制器先过滤不安全动作，再在安全集合中按预测 TTFT、显存释放量和迁移成本决策。若所有近似动作都不安全，则使用 full@CPU、restore 或 recompute，牺牲延迟而不牺牲质量 SLO。

这里有三个必须实现的细节：

1. 有损状态不能凭空 upgrade；只有保留 full shadow 或重新计算才能恢复精确状态。
2. 同 profile 内淘汰使用 LRU，不再设计 LPE score。
3. 用 hysteresis 和最小驻留时间避免 profile 反复切换。

**强度：中强；算法本身不是卖点，风险约束下的端到端收益才是卖点。**

## 4.4 C4：低审计漂移适配

部署中只抽取少量请求做 full-reference shadow execution，获得延迟到达的质量标签。AAL 用这些样本检测 coverage 漂移并更新校准阈值。

核心指标不是“在线学习更准”，而是：

- audit rate；
- 违例超标持续时间；
- 恢复到 \(\delta+3\)pp 所需 audited samples；
- audit 与 fallback 带来的系统开销。

**强度：中；作为 C2/C3 在非平稳边缘负载下成立的必要支撑。**

## 4.5 删除与降级项

| 旧内容 | 新状态 |
| --- | --- |
| LPE value-per-byte score | 删除；H1作为负结果归档 |
| H1 “比LRU降20%” | 删除，不再重跑同质 LPE |
| 质量-带宽相变面 | 删除核心定理身份；可作为成本热力图 |
| 在线 Pareto regret | 删除 |
| RRS 临界带宽定理 | 降为 controller 内的 exact fallback 规则 |
| COP | 重构为 QRP，仅保留对风险预测有用的特征 |
| TMS | 重构为 STC，创新来自 risk-safe admissible set |

## 4.6 创新门槛

| 结果 | 结论 |
| --- | --- |
| H0、H1、H2、H4全部通过 | 具备 ASPLOS/ATC/TC/TPDS 主线 |
| H0/H1通过，H2收益 10%–20% | 扩任务、模型和硬件；暂不投稿 |

<a id="五章-问题定义"></a>
# 五章 — 问题定义
---

## 5.1 系统对象

- 请求 \($i\in\mathcal I$\)：包含任务、prompt、RAG chunks、会话历史和到达时间。
- 缓存对象 \($o\in\mathcal O_i$\)：可复用 prefix/RAG/session KV。
- profile \($a\in\mathcal A$\)：对象的表示、放置和恢复方式。
- 任务组 \(g(i)\)：按任务类型与上下文长度预定义的 calibration group。

初始动作空间：

$[\mathcal A=\{\text{full@GPU}~,~\text{KIVI4@GPU}~,~\text{KIVI2@GPU}~,~\text{pruned50@GPU}~,~\text{full@CPU}~,~\text{recompute}\}]$

动作空间先小后大。H0/H1 阶段只需 full + 2种量化/剪枝；真实系统稳定后再加入组合 profile。

## 5.2 质量损失

对同一请求，full-KV 得分为 \(Q_i^{full}\)，采用动作 \(a\) 的得分为 \(Q_i^a\)。定义有界损失：

$[\ell_i(a)=\operatorname{clip}\left(\frac{Q_i^{full}-Q_i^a}{\max(|Q_i^{full}|,\tau)},0,1\right)]$

不同任务使用对应原始指标：

- QA/RAG：EM 或 token F1。
- 摘要：ROUGE-L。
- 代码：pass/fail 或单元测试通过率。
- 无标准答案的补充分析：full-output token agreement、logit KL；只能作为辅助，不能替代任务质量。

## 5.3 质量 SLO

给定可接受损失阈值 \(\varepsilon\) 和最大违例率 \(\delta\)：

$[R(a)=\Pr[\ell_i(a_i)>\varepsilon]\le\delta]$

同时报告：

- overall violation rate；
- 每任务组 \(R_g\)；
- q95/q99 loss；
- CVaR\(_{0.95}\)；
- worst-group violation。

主文不能只报告 overall coverage。

## 5.4 决策时可见信息

特征 \(x_i\) 必须在动作执行前可得：

$[x_i=[\text{task},n_{prompt},n_{chunk},n_{history},\text{reuse},\text{attention stats},\text{logit margin},\text{profile},\text{device state}]]$

禁止把生成完成后才能获得的质量标签、未来访问或 full-output 特征泄漏进 predictor。

## 5.5 风险预测与校准

损失预测器输出：

$[\hat\ell_i(a)=f_\theta(x_i,a)]$

校准器根据 calibration set 的残差构造上界。由于最终动作是从多个 profile 中自适应选择，需要给每个动作分配风险预算 ($\delta_a$\)，满足 ($\sum_a\delta_a\le\delta$)：

$[U_i(a)=\hat\ell_i(a)+q_{1-\delta_a}^{g(i),a}]$

安全动作集合：

$[\mathcal A_i^{safe}=\{a\in\mathcal A:U_i(a)\le\varepsilon\}$

若安全集合为空，强制加入质量精确的 `full@CPU` 或 `recompute`，因此质量约束有可行 fallback，但延迟可能升高。

## 5.6 资源与迁移约束

$\sum_o m(o,a_{o,t})\le M_t$，$a_{o,t}\rightarrow a_{o,t+1}\in\mathcal E_{valid}$

其中有效迁移图必须满足：

- full→quant/pruned 可以执行。
- quant/pruned→full 只有在 full shadow 存在或完成精确重算后才允许。
- CPU→GPU restore 计入带宽与反序列化成本。
- profile 切换计入量化/拷贝/同步成本。
- 每对象设置最小驻留时间 \(\Delta_{min}\) 与 hysteresis。

## 5.7 优化目标

在线控制器在安全动作中最小化：
				$\min_{\{a_{o,t}\}}\operatorname{CVaR}_{0.95}(TTFT_t)+\lambda_1 C_{mig}+\lambda_2 C_{audit}$

约束：
				$M_t\le M_t^{budget},\quad R_g\le\delta_g,\ \forall g,\quad a_{o,t}\rightarrow a_{o,t+1}\in\mathcal E_{valid}$

实验主比较采用固定 SLO，而不是调 \(\lambda\) 后比较无法解释的加权总分。

## 5.8 可写入论文的保证

在 calibration/test exchangeable、损失有界、分组预先固定的条件下，Conformal Guard 对相应总体或分组提供有限样本风险控制。为避免 adaptive profile selection 破坏 coverage，默认使用 simultaneous per-profile bounds；同时把“直接对完整策略做 conformal risk control”作为更紧的对照实现。论文需要证明：

1. 若各动作风险预算满足 \(\sum_a\delta_a\le\delta\)，且 STC 只从 \(\mathcal A_i^{safe}\) 选择动作，则通过 simultaneous coverage/union bound 得到端到端违例率上界。
2. exact fallback 的质量损失为零或可忽略时，安全集合始终可行。
3. 保证不覆盖未定义组、严重分布漂移或 audit 长期缺失；这些进入 limitation。

这部分称为“安全性质/命题”，不包装成原创 conformal 定理。

## 5.9 Oracle 与下界

- **Full oracle**：全程 full-KV，质量上界、资源成本下界反面。
- **Quality oracle**：知道真实 \(\ell_i(a)\)，在安全 profile 中选系统成本最低者。
- **Offline ILP**：小 trace 上求最优动作，衡量 STC optimality gap。
- **Belady/LRU**：仅用于缓存复用上界/标准换出，不再用于证明 LPE。

# 六章 — 算法设计
---

## 6.1 总体流程

```text
request / reusable KV object
        |
        v
QRP: extract decision-time features and predict loss for each profile
        |
        v
CG: conformal calibration -> per-profile upper bound U(x,a)
        |
        v
safe set A_safe = {a | U(x,a) <= epsilon}
        |
        v
STC: choose lowest-cost feasible action under GPU budget
        |
        +--> exact GPU / compressed GPU / pruned GPU
        +--> exact CPU restore
        +--> exact recompute
        |
        v
AAL: sample audits -> full reference -> coverage update / drift alarm
```

## 6.2 QRP：Quality Risk Profiler

### 输入特征

分三组逐步加入，必须做消融：

1. **零成本元数据**：任务、prompt 长度、历史长度、chunk 数、profile、复用次数。
2. **低成本模型信号**：部分层 attention entropy、top-k mass、首 token logit margin。
3. **系统状态**：当前显存、带宽、并发、缓存年龄；这些用于成本预测，不应直接冒充质量特征。

### 模型

先用 LightGBM/小 MLP，目标是可校准、低开销，不追求大模型 predictor。训练输出点损失$\hat\ell$或超阈值概率。

### 必须报告

- MAE/AUC 只能作为辅助。
- 核心是 calibration 后 coverage、bound slack 和安全 profile 可用率。
- 检查任务标签泄漏、长度 shortcut 和 profile imbalance。

## 6.3 CG：Conformal Guard

### 6.3.1 输入是什么

校准集必须包含每个请求在多个有损 profile 下的真实质量损失：

​			$\mathcal D_{cal}=\{(x_i~,~a~,~\hat\ell_i(a)~,~\ell_i(a)~,~g_i)\}$

其中：

- $x_i$：决策时可见特征。
- a：KIVI-4bit、KIVI-2bit、pruned50 等有损动作。
- $\hat\ell_i(a)$：QRP 预测的质量损失。
- $\ell_i(a)$：实际运行该 profile 后，相对 full-KV 的真实损失。
- $g_i$：任务类型 × 上下文长度档组成的 calibration group。

`full@GPU`、`full@CPU` 和精确 `recompute` 的质量损失视为0，直接进入安全集合，不参与有损动作的风险预算分配。

### 6.3.2 风险预算如何分配

系统级允许违例率为 $\delta$。若有 K 个有损动作，Pilot先采用等分：
					$\delta_a=\delta/K,\qquad \sum_{a=1}^{K}\delta_a=\delta$

这样可以通过 union bound 保证控制器从多个动作中自适应选择后，整体风险仍不超过 $\delta$。主实验再比较按动作使用频率或风险难度分配 $\delta_a$，但 Pilot 不先引入复杂优化。

### 6.3.3 校准分位数怎么计算

对每个 group-profile 对 (g,a)，计算单边 nonconformity residual：
					$s_i^{g,a}=\ell_i(a)-\hat\ell_i(a)$

设该组有 $n_{g,a}$ 个样本，残差从小到大排序后取：
					$	k=\left\lceil(n_{g,a}+1)(1-\delta_a)\right\rceil$ , $q_{g,a}=s_{k}^{g,a}$

若 $k>n_{g,a}$，说明样本量不足以支持当前风险水平。此时不能取最大值冒充保证，而是按以下层级扩大校准池：

```text
task x length x profile
    -> task x profile
    -> profile-global
    -> 将该有损动作标记为 unsafe
```

### 6.3.4 如何输出安全集合

新请求 x 到达后，对每个动作计算：
					$U_i(a)=\hat\ell_i(a)+q_{g(i),a}$$

然后输出：
					$\mathcal A_i^{safe}=$$\{a\in\mathcal A_{lossy}:U_i(a)\le\varepsilon\}
\cup
\mathcal A_{exact}$

CG输出的不是一个最终动作，而是一个候选集合。例如：

```text
Request group = qa-long, epsilon = 0.10

full@GPU:   exact                       -> safe
KIVI-4bit:  pred=0.03, q=0.04, U=0.07 -> safe
KIVI-2bit:  pred=0.09, q=0.05, U=0.14 -> unsafe
pruned50:   pred=0.06, q=0.03, U=0.09 -> safe
full@CPU:   exact                       -> safe

Output A_safe =
{full@GPU, KIVI-4bit, pruned50, full@CPU}
```

### 6.3.5 Pilot实现与对照

校准器不直接选动作，只输出安全集合。需要比较：

- naive per-profile：每个动作都直接使用 $\delta$，验证自适应选择后是否失去 coverage；
- simultaneous calibration：使用 $\delta_a$ 风险分配，作为 Pilot 默认算法；
- policy-level CRC：在 held-out trace 上直接校准完整策略，作为更紧但更复杂的主实验版本。

## 6.4 STC：Safe Tier Controller

### 6.4.1 输入与输出

输入：

- CG输出的每对象安全动作集合 $\mathcal A_o^{safe}$。
- 当前对象状态、GPU显存预算和需要释放的显存。
- 每个动作的显存、迁移、恢复、重算和未来复用成本预测。

输出：每个对象执行的具体动作。STC不能选择CG判定为unsafe的动作。

### 6.4.2 成本参数从哪里来

核心参数不使用人工拍定的固定值，而来自**真机微基准、在线测量和小型预测模型**：

| 参数                    | 计算方式                                                     |
| ----------------------- | ------------------------------------------------------------ |
| m(o,a)                  | 实测各profile的bytes/token × 对象token数 + metadata          |
| $T_{quant/prune}(o,a)$  | 按token数、batch、设备跑微基准，建立LUT或分段线性回归        |
| $T_{offload}$           | $size(o,a)/BW_{G2C}+d_{ser}$，带宽与固定开销真机测量         |
| $T_{restore}$           | $size(o,a)/BW_{C2G}+d_{deser}$                               |
| $T_{recompute}$         | 用实际prefill数据拟合 $f(n_{token}~,~batch~,~concurrency~,~device)$ |
| $T_{queue}$             | 根据queue depth、并发和service time做p95分位数回归           |
| $\hat p_{reuse}^{H}(o)$ | 用统一trace训练轻量survival/logistic模型；特征为recency、frequency、对象类型和age |
| 成本误差 $e_{95}$       | validation/在线日志中                                        |

$p_reuse$ 只用于估计未来是否会支付 restore/recompute 成本，不再用于设计新的换出分数；所有策略共用同一个 reuse estimator。

### 6.4.3 单动作成本

在未来 H 个请求窗口内：
			$\hat C_t(o,a)=T_{switch}(o,a) + \hat p_{reuse}^{H}(o)\left[T_{access}(o,a)+T_{queue}(o,a)\right]$

其中：

- GPU上的full/quant/pruned，$T_{access}$ 为读取、反量化或稀疏attention开销。
- `full@CPU`，$T_{access}=T_{restore}$。
- `drop/recompute`，$T_{access}=T_{recompute}$。
- $T_{switch}$ 是当前状态迁移到动作 a 的一次性成本。

### 6.4.4 显存不足时怎么选

需要释放的显存为：$R=M_{current}-M_{budget}$

对每个对象和安全动作计算：
			$\Delta M(o,a)=m(o,current)-m(o,a)$
			$\Delta C(o,a)=\hat C(o,a)-\hat C(o,current)\rho(o,a)=\frac{\max(\Delta C(o,a),0)+e_{95}(o,a)}{\Delta M(o,a)}$

$\rho$ 表示“每释放1MB显存带来的保守延迟代价”。Pilot采用：

```python
need_release = current_memory - memory_budget
candidates = build_safe_valid_transitions(resident_objects)

for transition in sorted(candidates, key=lambda x: x.rho):
    if transition.object_already_changed:
        continue
    apply(transition)
    need_release -= transition.delta_memory
    if need_release <= 0:
        break

if need_release > 0:
    apply_exact_fallback_with_lru_victim()
```

小trace上用multiple-choice knapsack ILP求离线最优，报告贪心算法的optimality gap。若gap长期超过10%，再考虑primal-dual或滚动窗口优化。

### 6.4.5 参数由谁决定

- 人工给定：用户SLO $\varepsilon$、$\delta$、显存预算 $M_{budget}$、最大audit开销。
- 真机测量：profile大小、带宽、量化/剪枝/恢复/重算时间。
- 数据拟合：prefill、queue、reuse和成本误差模型。
- 自动推导：风险分位数、\rho、动作排序和防抖阈值。

防抖阈值也不固定写死：
				$hysteresis(o,a)=e_{95}(o,a)+T_{switch}(o,a)$

只有预测累计收益超过该值才切换。最小驻留请求数由：
				$N_{min}=\left\lceil\frac{T_{switch}}{\max(\text{expected saving per request},\eta)}\right\rceil$

自动计算，并限制在合理区间内。

## 6.5 AAL：Audit & Adaptation Loop

### 6.5.1 “漂移”具体指什么

漂移指当前数据与建模/校准时期不再同分布：

| 漂移类型                 | 数学含义                                      | 例子                                               | 影响                         |
| ------------------------ | --------------------------------------------- | -------------------------------------------------- | ---------------------------- |
| workload/covariate drift | $P_t(x)\neq P_{cal}(x)$                       | QA比例变成代码任务；上下文从4K变32K；RAG chunk变多 | 某些校准组样本不足           |
| quality/concept drift    | $P_t(\ell\mid x,a)\neq P_{cal}(\ell\mid x,a)$ | 模型版本变化；新领域中KIVI-2bit更易出错            | CG的质量上界失准             |
| reuse drift              | $P_t(reuse\mid x)$变化                        | 热门知识突然变冷；会话模式改变                     | STC错误估计未来恢复/重算成本 |
| system-cost drift        | $P_t(T\mid state,a)$变化                      | PCIe带宽下降、并发升高、GPU降频、CPU繁忙           | STC动作排序失准              |

仅仅任务比例变化不一定需要报警；只有它导致coverage或成本预测残差持续超标，AAL才采取动作。

### 6.5.2 Audit如何获得真实质量

系统按 `(group, selected_profile)` 分层抽样：

- 初始 audit rate：5%。
- coverage 稳定后降到 1%–2%。
- 漂移告警后临时升到 10%，恢复后下降。
- 对被审计请求，同时保留当前profile输出并异步运行full-KV reference。
- 用任务指标计算真实 $\ell_i(a)$、violation和residual。
- 每个活跃group/profile保留最小audit配额，避免低频profile成为盲区。
- shadow执行的GPU时间、显存和能耗必须单独计量。

### 6.5.3 如何检测质量漂移

对系统整体和每个任务组 g 维护最近 W=100 个“最终被STC选中动作”的 audited 样本：
$\hat v_g=\frac{1}{W}\sum_{i=1}^{W}\mathbf 1[\ell_i(a_i)>\varepsilon]$

计算 violation rate 的 Wilson 区间。满足任一条件则触发 quality drift：

1. Wilson lower bound 连续两个窗口高于系统/分组目标 $\delta_g+\gamma$，Pilot取 $\gamma$=0.02。
2. 某个 profile 的 residual $s_i=\ell_i-\hat\ell_i$ 的 CUSUM/Page-Hinkley 超阈值，作为样本较少时的提前告警。
3. 新请求无法映射到已有有效 calibration group。

### 6.5.4 如何检测成本漂移

每次真实动作都能观察成本，无需额外full-KV audit：
				$e_t=\frac{|T_{actual}-T_{pred}|}{\max(T_{actual},1\text{ms})}$

当滚动窗口内 median $e_t$>15\%，或实测带宽相对profile值变化超过20%，触发 cost drift。这两个阈值在主实验中做敏感性分析。

### 6.5.5 漂移后更新什么

发生 quality drift：

1. 用近期 audited residual 更新 q_{g,a} 或 ACI 状态。
2. 暂时将失准动作移出 $\mathcal A^{safe}$。
3. 提高 exact fallback 和 audit rate。
4. 累积足够样本后重训 QRP。

发生 reuse/cost drift：

1. 更新带宽 EWMA 和微基准LUT修正项。
2. 在线更新 latency/reuse模型。
3. 重算所有候选动作的 \rho。
4. 不自动收紧质量阈值，除非同时观察到 quality drift。

当 Wilson upper bound 连续两个窗口低于 $\delta_g+\gamma$，且成本误差重新达标后解除告警，并逐步降低audit率。AAL最终输出更新后的 calibration quantile、cost model版本、unsafe profile名单和drift状态。

## 6.7 消融

- w/o conformal：只用点预测阈值。
- w/o group：只做全局校准。
- w/o audit：固定校准。
- w/o exact fallback：强制选最安全近似 profile。
- w/o hysteresis：观察迁移抖动。
- metadata-only / +attention / +logit：特征成本与收益。
- LRU replacement / engine default：确认普通换出不是贡献来源。

## 6.8 实现顺序

预实验直接实现与主实验相同的算法和接口：

1. 建立统一 `ProfileAdapter`，实际运行 full/KIVI/H2O并生成每请求×profile的质量、延迟、显存表。
2. 建立统一 `Policy` 接口，同时实现 TailGuardKV 与 full/static/utility/uncalibrated baseline。
3. 完成 QRP + simultaneous CG + STC + exact fallback 的完整决策闭环。
4. 用 measured-replay backend 跑一体化 Pilot，所有成本来自步骤1的实测。
5. 同期完成 `VLLMBackend`/`LMCacheBackend` 最小adapter和一条 smoke trace。
6. Pilot通过后，主实验只扩大配置矩阵并启用真实backend。
7. AAL是主实验阶段唯一新增的大模块，其余核心算法不得重写。

代码必须从第一天就区分 `Policy`、`Backend`、`MetricCollector` 和 `ExperimentConfig`。禁止预实验使用一次性 notebook，主实验再重新开发系统。

## 8.4 时间管理原则

- 新竞品每月检查一次，尤其关注 tail quality、risk calibration 和 adaptive KV compression。
- 任何核心门槛延误超过两周，必须在组会触发缩减或换题讨论。

# 十章 — 风险与应对措施

## 10.1 风险矩阵

| 风险 | 概率 | 影响 | 最早检测 | 应对 |
| --- | --- | --- | --- | --- |
| 平均质量与tail差异不明显 | 中 | 致命 | H0，2026-08-31 | 立即停止本方向 |
| conformal过度保守，几乎全full | 高 | 致命 | H1，2026-09-15 | 改特征/分组；两轮失败换题 |
| workload shift下coverage崩溃 | 高 | 高 | H3 | audit/ACI；仍失败则取消动态保证 |
| AdaptCache/KVServe加入tail guarantee | 中 | 致命 | 每月文献检查 | 重新差异审计，必要时换题 |
| 质量标签成本过高 | 高 | 高 | H0/H1 | 分层采样、少量profile、异步audit |
| 真实kernel不支持逐对象profile | 高 | 高 | vLLM接入期 | 限制为请求/segment粒度并诚实说明 |
| controller收益被迁移开销抵消 | 中 | 高 | H2 | hysteresis、批量迁移、减少动作空间 |
| 端侧设备趋势不一致 | 中 | 高 | H4 | 单独建成本模型；不能只报服务器 |
| baseline复现不完整 | 中 | 高 | E3前 | 明确“inspired”实现和差异，不冒充原系统 |
| 只剩统计方法、系统贡献弱 | 中 | 高 | P2 | 强化真实引擎、fallback和shift闭环 |

## 10.2 核心退出条件

以下任一成立即停止以 CCF-A 为目标继续投入：

- H0不存在稳定 tail-risk。
- H1只能通过全部 exact/full满足违例率。
- H2对最佳 baseline 的收益低于10%，且 Oracle gap 也小。
- H4真实系统无法实现或开销超过收益。
- 最新竞品已提供相同粒度的 calibrated tail SLO 与在线闭环。

## 10.3 Fallback

### F1：静态校准版本

仅当 H0/H1通过、H3失败时采用。保留 group conformal 和安全 controller，删除在线漂移保证。投稿目标下调，不把它包装成完整 CCF-A 主线。

### F2：Tail-quality benchmark

如果系统收益不足但 H0数据很强，可整理 benchmark、failure taxonomy 和评测协议。它可形成独立研究资产，但不能默认达到 CCF-A。

### F3：换题

如果 H0或H1失败，优先换到新的可证伪问题，不再退回 EdgeKVLife/LPE。旧的“纯 lifecycle fallback”已经被 H1否定。

## 10.4 明确不做的补救

- 不通过调更多 LPE 参数制造局部胜点。
- 不把平均质量预算改个名字继续声称 guarantee。
- 不用纯仿真替代真实系统后仍投系统类 CCF-A。
- 不为了保留理论而强行加入与系统无关的 regret 证明。
