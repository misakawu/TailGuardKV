# Pilot实验提取

## 0.7 一体化预实验：直接实现算法并与 baseline 比较

预实验不是单独的问题验证脚本，而是**缩小规模的主实验**。必须直接交付可继续扩展的 TailGuardKV 最小完整原型：

### 0.7.1 一次性实现的代码骨架

```text
tailguardkv/
  profiles/        # full、KIVI、H2O/SnapKV、exact-offload
  policies/        # TailGuardKV、static、utility、uncalibrated
  calibration/     # predictor、simultaneous conformal、policy-level CRC
  backends/        # measured-replay backend + vLLM/LMCache adapter
  metrics/         # quality、coverage、TTFT、memory、overhead
  configs/         # pilot/main 共用，主实验只扩大参数网格
  run_experiment.py
```

所有策略统一实现：

```python
class Policy:
    def decide(self, request, cache_state, device_state) -> Action:
        ...
```

baseline 与 TailGuardKV 必须经过同一 backend、同一 trace、同一指标代码，禁止为不同策略写不同实验脚本。

### 0.7.2 Pilot 配置

| 项目     | 最小配置                                                   |
| -------- | ---------------------------------------------------------- |
| 模型     | Qwen2.5-7B-Instruct                                        |
| 任务     | 1个 + 摘要/代码任选1个；每类先200–300请求                  |
| Profile  | full、KIVI-4bit、KIVI-2bit、H2O/SnapKV一档、exact fallback |
| 显存预算 | 2个真正触发profile选择的紧预算                             |
| 质量SLO  | $\varepsilon$ 2档，$\delta\in\{0.05,0.10\}$                |
| 重复     | 系统回放每cell至少3次                                      |

质量 loss、profile latency 和 memory 必须来自实际模型/profile实测，不能使用拍脑袋的解析常数。预实验可以先用 measured-replay backend 做大网格，但必须完成一条 vLLM/LMCache 小 trace 的 smoke test，保证接口能落到真实引擎。

### 0.7.3 必须同时实现的 baseline

1. `full_lru`：全精度 + 引擎默认换出。
2. `static_best`：离线选择满足平均质量约束的最佳固定 profile。
3. `static_safe`：满足 tail SLO 的最保守固定 profile。
4. `utility_dynamic`：AdaptCache 风格平均 quality-utility 控制器。
5. `uncalibrated_dynamic`：使用点预测阈值、不做 conformal。
6. `tailguard`：QRP + simultaneous/policy-level calibration + STC + exact fallback。
7. `quality_oracle`：仅作上界，不参与“最佳可部署 baseline”排名。

### 0.7.4 一次运行同时回答 H0/H1/H2-lite

同一批 pilot 结果必须输出：

- H0：mean/p95/p99/CVaR/violation，证明 tail 问题。
- H1：target-vs-empirical coverage、worst-group、UCB slack、exact fallback ratio。
- H2-lite：固定 ($\varepsilon,\delta$)下各策略的 p95 TTFT、显存和动作分布。
- 系统开销：predictor、calibration、controller 和迁移时间。

### 0.7.5 Pilot 通过标准

必须同时满足：

1. H0 tail-risk 成立。
2. $\delta=5\%$ 时 empirical violation ≤8%，且 exact fallback ratio <60%，避免“全部full”的空保证。
3. 至少2个有效cell中，相比最佳可部署 baseline，p95 TTFT降≥10%或显存降≥15%。
4. controller开销 <5% TTFT。
5. `run_experiment.py` 只修改配置即可扩展到主实验。

通过后，主实验只做四件事：增加模型、增加任务、增加预算/SLO网格、切换到真实 vLLM/LMCache/端侧 backend。算法和 baseline 不得推倒重写。

# 算法设计

---

## 6.1 总体流程

```text
request / reusable KV object
        |
        v
QRP: extract decision-time features and predict loss for each profile
        |
        v
CG: conformal calibration -> per-profile upper bound U(x,a)
        |
        v
safe set A_safe = {a | U(x,a) <= epsilon}
        |
        v
STC: choose lowest-cost feasible action under GPU budget
        |
        +--> exact GPU / compressed GPU / pruned GPU
        +--> exact CPU restore
        +--> exact recompute
        |
        v
AAL: sample audits -> full reference -> coverage update / drift alarm
```

## 6.2 QRP：Quality Risk Profiler

### 输入特征

分三组逐步加入，必须做消融：

1. **零成本元数据**：任务、prompt 长度、历史长度、chunk 数、profile、复用次数。
2. **低成本模型信号**：部分层 attention entropy、top-k mass、首 token logit margin。
3. **系统状态**：当前显存、带宽、并发、缓存年龄；这些用于成本预测，不应直接冒充质量特征。

### 模型

先用 LightGBM/小 MLP，目标是可校准、低开销，不追求大模型 predictor。训练输出点损失$\hat\ell$或超阈值概率。

### 必须报告

- MAE/AUC 只能作为辅助。
- 核心是 calibration 后 coverage、bound slack 和安全 profile 可用率。
- 检查任务标签泄漏、长度 shortcut 和 profile imbalance。

## 6.3 CG：Conformal Guard

### 6.3.1 输入是什么

校准集必须包含每个请求在多个有损 profile 下的真实质量损失：

​			$\mathcal D_{cal}=\{(x_i~,~a~,~\hat\ell_i(a)~,~\ell_i(a)~,~g_i)\}$

其中：

- $x_i$：决策时可见特征。
- a：KIVI-4bit、KIVI-2bit、pruned50 等有损动作。
- $\hat\ell_i(a)$：QRP 预测的质量损失。
- $\ell_i(a)$：实际运行该 profile 后，相对 full-KV 的真实损失。
- $g_i$：任务类型 × 上下文长度档组成的 calibration group。

`full@GPU`、`full@CPU` 和精确 `recompute` 的质量损失视为0，直接进入安全集合，不参与有损动作的风险预算分配。

### 6.3.2 风险预算如何分配

系统级允许违例率为 $\delta$。若有 K 个有损动作，Pilot先采用等分：
					$\delta_a=\delta/K,\qquad \sum_{a=1}^{K}\delta_a=\delta$

这样可以通过 union bound 保证控制器从多个动作中自适应选择后，整体风险仍不超过 $\delta$。主实验再比较按动作使用频率或风险难度分配 $\delta_a$，但 Pilot 不先引入复杂优化。

### 6.3.3 校准分位数怎么计算

对每个 group-profile 对 (g,a)，计算单边 nonconformity residual：
					$s_i^{g,a}=\ell_i(a)-\hat\ell_i(a)$

设该组有 $n_{g,a}$ 个样本，残差从小到大排序后取：
					$	k=\left\lceil(n_{g,a}+1)(1-\delta_a)\right\rceil$ , $q_{g,a}=s_{k}^{g,a}$

若 $k>n_{g,a}$，说明样本量不足以支持当前风险水平。此时不能取最大值冒充保证，而是按以下层级扩大校准池：

```text
task x length x profile
    -> task x profile
    -> profile-global
    -> 将该有损动作标记为 unsafe
```

### 6.3.4 如何输出安全集合

新请求 x 到达后，对每个动作计算：
					$U_i(a)=\hat\ell_i(a)+q_{g(i),a}$$

然后输出：
					$\mathcal A_i^{safe}=$$\{a\in\mathcal A_{lossy}:U_i(a)\le\varepsilon\}
\cup
\mathcal A_{exact}$

CG输出的不是一个最终动作，而是一个候选集合。例如：

```text
Request group = qa-long, epsilon = 0.10

full@GPU:   exact                       -> safe
KIVI-4bit:  pred=0.03, q=0.04, U=0.07 -> safe
KIVI-2bit:  pred=0.09, q=0.05, U=0.14 -> unsafe
pruned50:   pred=0.06, q=0.03, U=0.09 -> safe
full@CPU:   exact                       -> safe

Output A_safe =
{full@GPU, KIVI-4bit, pruned50, full@CPU}
```

### 6.3.5 Pilot实现与对照

校准器不直接选动作，只输出安全集合。需要比较：

- naive per-profile：每个动作都直接使用 $\delta$，验证自适应选择后是否失去 coverage；
- simultaneous calibration：使用 $\delta_a$ 风险分配，作为 Pilot 默认算法；
- policy-level CRC：在 held-out trace 上直接校准完整策略，作为更紧但更复杂的主实验版本。

## 6.4 STC：Safe Tier Controller

### 6.4.1 输入与输出

输入：

- CG输出的每对象安全动作集合 $\mathcal A_o^{safe}$。
- 当前对象状态、GPU显存预算和需要释放的显存。
- 每个动作的显存、迁移、恢复、重算和未来复用成本预测。

输出：每个对象执行的具体动作。STC不能选择CG判定为unsafe的动作。

### 6.4.2 成本参数从哪里来

核心参数不使用人工拍定的固定值，而来自**真机微基准、在线测量和小型预测模型**：

| 参数                    | 计算方式                                                     |
| ----------------------- | ------------------------------------------------------------ |
| m(o,a)                  | 实测各profile的bytes/token × 对象token数 + metadata          |
| $T_{quant/prune}(o,a)$  | 按token数、batch、设备跑微基准，建立LUT或分段线性回归        |
| $T_{offload}$           | $size(o,a)/BW_{G2C}+d_{ser}$，带宽与固定开销真机测量         |
| $T_{restore}$           | $size(o,a)/BW_{C2G}+d_{deser}$                               |
| $T_{recompute}$         | 用实际prefill数据拟合 $f(n_{token}~,~batch~,~concurrency~,~device)$ |
| $T_{queue}$             | 根据queue depth、并发和service time做p95分位数回归           |
| $\hat p_{reuse}^{H}(o)$ | 用统一trace训练轻量survival/logistic模型；特征为recency、frequency、对象类型和age |
| 成本误差 $e_{95}$       | validation/在线日志中                                        |

$p_reuse$ 只用于估计未来是否会支付 restore/recompute 成本，不再用于设计新的换出分数；所有策略共用同一个 reuse estimator。

### 6.4.3 单动作成本

在未来 H 个请求窗口内：
			$\hat C_t(o,a)=T_{switch}(o,a) + \hat p_{reuse}^{H}(o)\left[T_{access}(o,a)+T_{queue}(o,a)\right]$

其中：

- GPU上的full/quant/pruned，$T_{access}$ 为读取、反量化或稀疏attention开销。
- `full@CPU`，$T_{access}=T_{restore}$。
- `drop/recompute`，$T_{access}=T_{recompute}$。
- $T_{switch}$ 是当前状态迁移到动作 a 的一次性成本。

### 6.4.4 显存不足时怎么选

需要释放的显存为：$R=M_{current}-M_{budget}$

对每个对象和安全动作计算：
			$\Delta M(o,a)=m(o,current)-m(o,a)$
			$\Delta C(o,a)=\hat C(o,a)-\hat C(o,current)\rho(o,a)=\frac{\max(\Delta C(o,a),0)+e_{95}(o,a)}{\Delta M(o,a)}$

$\rho$ 表示“每释放1MB显存带来的保守延迟代价”。Pilot采用：

```python
need_release = current_memory - memory_budget
candidates = build_safe_valid_transitions(resident_objects)

for transition in sorted(candidates, key=lambda x: x.rho):
    if transition.object_already_changed:
        continue
    apply(transition)
    need_release -= transition.delta_memory
    if need_release <= 0:
        break

if need_release > 0:
    apply_exact_fallback_with_lru_victim()
```

小trace上用multiple-choice knapsack ILP求离线最优，报告贪心算法的optimality gap。若gap长期超过10%，再考虑primal-dual或滚动窗口优化。

### 6.4.5 参数由谁决定

- 人工给定：用户SLO $\varepsilon$、$\delta$、显存预算 $M_{budget}$、最大audit开销。
- 真机测量：profile大小、带宽、量化/剪枝/恢复/重算时间。
- 数据拟合：prefill、queue、reuse和成本误差模型。
- 自动推导：风险分位数、\rho、动作排序和防抖阈值。

防抖阈值也不固定写死：
				$hysteresis(o,a)=e_{95}(o,a)+T_{switch}(o,a)$

只有预测累计收益超过该值才切换。最小驻留请求数由：
				$N_{min}=\left\lceil\frac{T_{switch}}{\max(\text{expected saving per request},\eta)}\right\rceil$

自动计算，并限制在合理区间内。

## 6.5 AAL：Audit & Adaptation Loop

### 6.5.1 “漂移”具体指什么

漂移指当前数据与建模/校准时期不再同分布：

| 漂移类型                 | 数学含义                                      | 例子                                               | 影响                         |
| ------------------------ | --------------------------------------------- | -------------------------------------------------- | ---------------------------- |
| workload/covariate drift | $P_t(x)\neq P_{cal}(x)$                       | QA比例变成代码任务；上下文从4K变32K；RAG chunk变多 | 某些校准组样本不足           |
| quality/concept drift    | $P_t(\ell\mid x,a)\neq P_{cal}(\ell\mid x,a)$ | 模型版本变化；新领域中KIVI-2bit更易出错            | CG的质量上界失准             |
| reuse drift              | $P_t(reuse\mid x)$变化                        | 热门知识突然变冷；会话模式改变                     | STC错误估计未来恢复/重算成本 |
| system-cost drift        | $P_t(T\mid state,a)$变化                      | PCIe带宽下降、并发升高、GPU降频、CPU繁忙           | STC动作排序失准              |

仅仅任务比例变化不一定需要报警；只有它导致coverage或成本预测残差持续超标，AAL才采取动作。

### 6.5.2 Audit如何获得真实质量

系统按 `(group, selected_profile)` 分层抽样：

- 初始 audit rate：5%。
- coverage 稳定后降到 1%–2%。
- 漂移告警后临时升到 10%，恢复后下降。
- 对被审计请求，同时保留当前profile输出并异步运行full-KV reference。
- 用任务指标计算真实 $\ell_i(a)$、violation和residual。
- 每个活跃group/profile保留最小audit配额，避免低频profile成为盲区。
- shadow执行的GPU时间、显存和能耗必须单独计量。

### 6.5.3 如何检测质量漂移

对系统整体和每个任务组 g 维护最近 W=100 个“最终被STC选中动作”的 audited 样本：
$\hat v_g=\frac{1}{W}\sum_{i=1}^{W}\mathbf 1[\ell_i(a_i)>\varepsilon]$

计算 violation rate 的 Wilson 区间。满足任一条件则触发 quality drift：

1. Wilson lower bound 连续两个窗口高于系统/分组目标 $\delta_g+\gamma$，Pilot取 $\gamma$=0.02。
2. 某个 profile 的 residual $s_i=\ell_i-\hat\ell_i$ 的 CUSUM/Page-Hinkley 超阈值，作为样本较少时的提前告警。
3. 新请求无法映射到已有有效 calibration group。

### 6.5.4 如何检测成本漂移

每次真实动作都能观察成本，无需额外full-KV audit：
				$e_t=\frac{|T_{actual}-T_{pred}|}{\max(T_{actual},1\text{ms})}$

当滚动窗口内 median $e_t$>15\%，或实测带宽相对profile值变化超过20%，触发 cost drift。这两个阈值在主实验中做敏感性分析。

### 6.5.5 漂移后更新什么

发生 quality drift：

1. 用近期 audited residual 更新 q_{g,a} 或 ACI 状态。
2. 暂时将失准动作移出 $\mathcal A^{safe}$。
3. 提高 exact fallback 和 audit rate。
4. 累积足够样本后重训 QRP。

发生 reuse/cost drift：

1. 更新带宽 EWMA 和微基准LUT修正项。
2. 在线更新 latency/reuse模型。
3. 重算所有候选动作的 \rho。
4. 不自动收紧质量阈值，除非同时观察到 quality drift。

当 Wilson upper bound 连续两个窗口低于 $\delta_g+\gamma$，且成本误差重新达标后解除告警，并逐步降低audit率。AAL最终输出更新后的 calibration quantile、cost model版本、unsafe profile名单和drift状态。

## 6.7 消融

- w/o conformal：只用点预测阈值。
- w/o group：只做全局校准。
- w/o audit：固定校准。
- w/o exact fallback：强制选最安全近似 profile。
- w/o hysteresis：观察迁移抖动。
- metadata-only / +attention / +logit：特征成本与收益。
- LRU replacement / engine default：确认普通换出不是贡献来源。

## 6.8 实现顺序

预实验直接实现与主实验相同的算法和接口：

1. 建立统一 `ProfileAdapter`，实际运行 full/KIVI/H2O并生成每请求×profile的质量、延迟、显存表。
2. 建立统一 `Policy` 接口，同时实现 TailGuardKV 与 full/static/utility/uncalibrated baseline。
3. 完成 QRP + simultaneous CG + STC + exact fallback 的完整决策闭环。
4. 用 measured-replay backend 跑一体化 Pilot，所有成本来自步骤1的实测。
5. 同期完成 `VLLMBackend`/`LMCacheBackend` 最小adapter和一条 smoke trace。
6. Pilot通过后，主实验只扩大配置矩阵并启用真实backend。
7. AAL是主实验阶段唯一新增的大模块，其余核心算法不得重写。

代码必须从第一天就区分 `Policy`、`Backend`、`MetricCollector` 和 `ExperimentConfig`。禁止预实验使用一次性 notebook，主实验再重新开发系统。
